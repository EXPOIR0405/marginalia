## 2026-05-29 — 어드민 시스템 구축

### 배경

매번 MDX 파일을 직접 만드는 게 너무 번거로워서 사이트 내부에서 바로 글을 쓰고 저장할 수 있는 어드민 시스템을 만들기로 했다. Vercel에 배포된 환경이라 파일시스템에 직접 쓸 수 없어서, GitHub API로 MDX를 레포에 커밋하는 방식을 선택했다. 이미지는 Supabase Storage 무료 티어를 쓰기로.

---

### ✅ 완료된 것들

**기반 설정**
- `@supabase/supabase-js` 설치
- `.env.local`에 환경변수 추가: `ADMIN_PASSWORD`, `GITHUB_TOKEN`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `OPENAI_API_KEY`
- `next.config.ts`에 Supabase 이미지 도메인(`**.supabase.co`) 추가

**인증**
- `middleware.ts` — `/admin/editor` 접근 시 `admin_auth` 쿠키 검사, 없으면 `/admin`으로 리다이렉트
- `/admin` 로그인 페이지 — 비밀번호 입력 → `/api/admin/auth` → httpOnly 쿠키 발급 (7일)

**에디터 (`/admin/editor`)**
- 왼쪽 사이드바: 슬러그, 날짜, 시리즈, EP 번호, 한줄 소개, 태그, 썸네일 이미지 업로드
- 오른쪽: 제목 입력 + MDX 텍스트에디터 + 글자수/읽기 시간

**API 라우트**
- `POST /api/admin/auth` — 비밀번호 검증 후 쿠키 설정
- `POST /api/admin/writings` — frontmatter 조립 → base64 인코딩 → GitHub API (`PUT /repos/.../contents/...`)로 MDX 파일 커밋
- `POST /api/admin/upload` — Supabase Storage `writings-images` 버킷에 이미지 업로드, public URL 반환

**AI 다듬기**
- `POST /api/admin/refine` — OpenAI GPT-4o 사용
- 단순 교정이 아니라 "작가처럼 재밌게, 문장 리듬 살리기, 저자 목소리 유지" 수준의 프롬프트로 재작성
- 전후 비교 모달(좌: 원본 / 우: AI 버전), AI 버전 직접 수정 가능, [적용]/[취소]

**소재 추천**
- `POST /api/admin/suggest-topics` — OpenAI GPT-4o로 연재 시리즈 맥락에 맞는 소재 5개 제안
- 클릭하면 제목 입력란에 자동 삽입
- ❗ 초기 버그: `response_format: { type: 'json_object' }`와 배열 직접 반환 요청이 충돌해서 파싱 실패 → `{"topics": [...]}` 형태로 수정 후 해결

### ❌ 미해결 — 문장 제안 기능

**의도한 동작**: ON/OFF 토글을 켜두면 타이핑 멈춘 지 4초 후 AI가 다음 문장 1개를 에디터 하단 바에 제안. [삽입]을 누르면 본문 끝에 추가, [무시]를 누르면 사라짐. API 토큰을 쓰는 기능이라 토글로 제어 가능하게 설계.

---

#### 트러블슈팅 과정

**1단계 — 일단 기다려봤다: 4초가 지나도 아무것도 안 뜸**

토글을 켜고 글을 쓴 뒤 타이핑을 멈췄다. 4초가 지나도 에디터 하단에 아무것도 나타나지 않았다. 에러 메시지도 없었다. 처음엔 API 호출 자체가 안 되는 줄 알았다.

---

**2단계 — API 에러인가: 에러 상태 추가**

당시 코드는 API 실패 시 조용히 넘어가고 있었다. 원인을 보기 위해 `suggestionError` state를 추가하고, fetch 실패 시 에러 메시지를 suggestion 바에 표시하도록 수정했다.

```tsx
// 수정 전 — 에러 시 아무것도 안 함
if (res.ok) setSentenceSuggestion(data.sentence)

// 수정 후 — 에러 표시
if (res.ok && data.sentence) {
  setSentenceSuggestion(data.sentence)
} else {
  setSuggestionError(data.error ?? 'API 오류')
}
```

서버 재시작 후 다시 시도했지만 여전히 아무것도 나타나지 않았다. 에러 메시지도 없었다.

---

**3단계 — API 자체가 안 불리는 건가: useCallback stale closure 발견**

당시 fetch 함수는 이렇게 되어 있었다:

```typescript
const fetchSentenceSuggestion = useCallback(async (currentContent: string) => {
  setLoadingSuggestion(true)
  const res = await fetch('/api/admin/suggest-sentence', { ... })
  const data = await res.json()
  if (res.ok) setSentenceSuggestion(data.sentence)
  setLoadingSuggestion(false)
}, [])  // ← 빈 의존성 배열
```

문제는 React 18 Strict Mode의 마운트 동작이었다. Strict Mode에서는 개발 환경에서 컴포넌트를 **마운트 → 언마운트 → 리마운트** 한다. `useCallback`은 빈 의존성 배열로 함수를 한 번만 만들고 캐시한다.

그런데 이 함수가 캐시될 때 클로저 안에 갇히는 `setSentenceSuggestion`은 **첫 번째 마운트**에서 만들어진 setter다. 언마운트 이후엔 이 컴포넌트 인스턴스의 state가 파괴된다. 두 번째 마운트에서 새 setter가 생성되지만, `useCallback`은 여전히 첫 번째 것을 붙들고 있다. 결국 이미 죽은 state에 값을 쓰고 있었던 것이다. React 18에서는 언마운트된 컴포넌트의 setState 호출은 경고도 없이 그냥 무시된다.

→ `useCallback` 제거, fetch 로직을 `useEffect` 안으로 직접 이동. 각 effect 실행은 자신만의 클로저를 만들고, cleanup 플래그(`cancelled`)로 언마운트 후 state 업데이트를 막는다.

```typescript
useEffect(() => {
  if (!suggestionEnabled || !content.trim()) return

  let cancelled = false
  const timer = setTimeout(async () => {
    if (cancelled) return
    // fetch...
    if (!cancelled) setSentenceSuggestion(data.sentence)
  }, 4000)

  return () => { cancelled = true; clearTimeout(timer) }
}, [content, suggestionEnabled])
```

---

**4단계 — 고쳤는데 여전히 안 보임: API 응답 확인**

수정 후 재시도. 서버 터미널에서 이게 찍혔다:

```
POST /api/admin/suggest-sentence 200 in 1467ms
```

API는 200으로 정상 응답하고 있다. 그러면 클라이언트에서 받은 데이터가 이상한 건가? `console.log`를 추가했다.

```typescript
console.log('>>> setSentenceSuggestion:', data.sentence)
setSentenceSuggestion(data.sentence)
```

브라우저 콘솔 출력:

```
>>> setSentenceSuggestion: 민정이와의 소중한 추억들을 계속 쌓아가고 싶어.
```

`setSentenceSuggestion`은 올바른 값으로 **실제로 호출되고 있었다.** React state에 값이 들어가야 하는데 화면엔 아무것도 없었다.

---

**5단계 — state는 세팅되는데 왜 안 보이지: 직접 화면에 출력**

state가 세팅되는데 렌더링이 안 되는 건지 확인하기 위해, 조건부 렌더링 없이 글자수 바에 state 값을 직접 출력했다.

```tsx
<div className="px-8 py-2 border-t border-gray-50 flex items-center gap-4">
  <p className="text-xs text-gray-400">{content.length.toLocaleString()}자</p>
  {/* 디버그 */}
  <p className="text-xs text-red-400 ml-auto">
    [dbg] enabled:{String(suggestionEnabled)} sentence:{sentenceSuggestion.slice(0,10)||'(없음)'}
  </p>
</div>
```

이 빨간 디버그 텍스트가 **보이지 않았다.** 글자수 바 자체는 평소에 잘 보이는데, `setSentenceSuggestion`이 호출된 이후로는 그게 사라졌다.

→ **React state 문제가 아니라 CSS 레이아웃 문제**라는 결론에 도달.

---

**6단계 — 원인 파악: overflow-hidden + flex-1 + min-h-0 부재**

에디터 오른쪽 컬럼의 구조:

```
<div class="flex-1 flex flex-col overflow-hidden">   ← 컨테이너
  <div>제목 입력 (~60px)</div>
  <textarea class="flex-1 ...">본문</textarea>        ← 남은 공간 전부 차지
  <div>suggestion 바 (조건부, ~44px)</div>            ← 여기가 문제
  <div>글자수 바 (~30px)</div>
</div>
```

`textarea`에 `flex-1`이 있어서 컨테이너의 남은 공간을 **전부** 채운다. 초기 상태(suggestion 바 없음)에선 컨테이너 높이 = 제목(60) + textarea(나머지) + 글자수(30). 딱 맞는다.

suggestion 바(44px)가 새로 나타나면 총 콘텐츠 높이가 44px 초과한다. 이때 textarea가 줄어들어야 하는데, **`min-h-0`이 없으면 줄어들지 않는다.** CSS flex 아이템의 기본 `min-height`는 `auto`로, 자신의 콘텐츠 높이 아래로는 못 줄어든다는 뜻이다. 결국 suggestion 바는 컨테이너 밖으로 밀려나고, `overflow-hidden`이 잘라버린다. 바로 그 아래 글자수 바까지 같이 잘렸기 때문에 [dbg]도 보이지 않았던 것이다.

→ `<textarea>`에 `min-h-0` 클래스 추가.

```tsx
// 수정 전
className="flex-1 p-8 font-mono ..."

// 수정 후
className="flex-1 min-h-0 p-8 font-mono ..."
```

### 메모

- Supabase Storage는 public 버킷 + service_role 키 조합으로 쓰면 RLS 설정 없이 심플하게 운영 가능. 개인 블로그 수준엔 무료 티어(1GB)로 충분.
- GitHub API로 MDX 파일을 커밋하는 방식은 기존 MDX 기반 구조를 그대로 유지할 수 있어서 깔끔했다. 이미 있는 파일에 같은 slug로 저장하면 충돌 에러가 나는데, 추후 "글 수정" 기능 만들 때 SHA 값 받아서 update하는 로직 추가 필요.
